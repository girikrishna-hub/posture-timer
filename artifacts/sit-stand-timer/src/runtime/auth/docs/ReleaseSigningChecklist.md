# Release Signing Checklist — Android Production Deployment

**Status:** Template — populate with actual values before first production release.  
**Updated by:** Engineering team before each release.

---

## CRITICAL: Most production-only auth failures originate from signing mismatches

Google OAuth Android clients are locked to a specific SHA-1 fingerprint.  
If the production APK is signed with a different key than what's registered,
`GoogleSignIn.signIn()` will silently fail with error code 10 (`DEVELOPER_ERROR`).

The `AuthProviderError` in this runtime classifies code 10 as `INVALID_TOKEN` —
the user will be prompted to re-authenticate with an error. But the **real fix**
is ensuring signing configuration is correct before deployment.

## 6. Pre-Deployment Verification Steps

Run these steps before each production release:

```bash
# 1. Get production signing certificate fingerprint
keytool -list -v -keystore release.keystore -alias [alias] | grep SHA

# 2. Verify it matches Google Cloud Console OAuth client
# → Open: https://console.cloud.google.com/apis/credentials
# → Find your Android OAuth client
# → Confirm SHA-1 matches step 1

# 3. Build release APK and verify signing
./gradlew bundleRelease
apksigner verify --verbose release.aab

# 4. Test Google Sign-In on a physical device with production APK
# → Should NOT get DEVELOPER_ERROR (code 10)
# → Should complete sign-in flow

# 5. Verify Clerk token exchange succeeds
# → Check AuthDiagnosticsJournal for AUTH_SIGN_IN_SUCCEEDED
# → Confirm no AUTH_SIGN_IN_FAILED with "token_invalid"
```

## 7. Auth Flow Smoke Test (Post-Signing Verification)

On each new signed build, before wider release:

- [ ] Cold start → Google Sign-In → `FSM=SIGNED_IN` within 3s
- [ ] Kill and restart → vault restore → `FSM=SIGNED_IN` within 2s
- [ ] Disable network → restart → `FSM=OFFLINE_RECOVERY`
- [ ] Re-enable network → `FSM=SIGNED_IN` (auto-recovery)
- [ ] Revoke app access in Google Account settings → next sign-in attempt shows re-auth UI (not infinite loop)

## 8. Notes on `allowBackup`

Ensure `android/app/src/main/AndroidManifest.xml` has:

```xml
<application
  android:allowBackup="false"
  ...>
```

Without this, ADB backup can expose SharedPreferences vault data to an attacker
with physical device access. This is particularly important because SharedPreferences
is not encrypted in the current implementation.
