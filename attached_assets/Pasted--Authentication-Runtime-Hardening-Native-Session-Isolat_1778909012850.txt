# Authentication Runtime Hardening & Native Session Isolation Prompt

IMPORTANT:

This is NOT a rewrite request.

The RuntimeCore architecture already exists and must be preserved.

The goal now is:

- eliminate remaining hybrid web-auth behavior
- harden native runtime guarantees
- remove Clerk frontend operational ownership
- make auth restoration deterministic
- make refresh/runtime recovery fully native-safe
- eliminate lifecycle race conditions
- eliminate hidden session desynchronization
- make failures diagnosable

This phase is:
runtime hardening and auth isolation.

Do NOT regress the architecture back toward:
- React-owned auth
- browser session semantics
- redirect-driven auth
- WebView session dependence

---

# CURRENT ARCHITECTURE STATUS

The following systems already exist and should be preserved:

- AuthStateMachine
- AuthStateStore
- AuthOperationQueue
- RuntimeBootBarrier
- GoogleAuthAdapter
- ClerkBridgeAdapter
- AuthSessionManager
- AuthLifecycleCoordinator
- AuthRecoveryCoordinator
- AuthCapabilityRegistry
- AuthDiagnosticsJournal
- useAuthRuntime
- useBootBarrier
- NativeAuthScreen
- NativeGoogleSignInButton
- AuthRuntimeOverlay

The runtime already:
- boots before React renders
- uses native Google Sign-In
- serializes auth operations
- restores persisted sessions
- gates UI behind boot restoration
- uses a runtime FSM
- uses lifecycle coordinators

DO NOT replace these systems.

Instead:
stabilize and isolate them.

---

# PRIMARY OBJECTIVE

Eliminate ALL remaining operational dependence on:

- Clerk React hooks
- browser session semantics
- WebView persistence
- React lifecycle timing
- frontend Clerk session ownership

The runtime itself must become:
the sole operational auth authority.

---

# CRITICAL ISSUE: CLERK FRONTEND OWNERSHIP LEAKAGE

Current architecture still includes:

ClerkRuntimeBridge
→ useSignIn()
→ useAuth()
→ useClerk()

This creates hidden hybrid ownership.

This must be eliminated.

IMPORTANT:

React hooks may observe auth state.

They must NOT:
- own session continuity
- own refresh capability
- own restoration authority
- own auth validity
- own session truth

---

# REQUIRED REFACTOR

Refactor ClerkBridgeAdapter into a pure infrastructure adapter.

Create:

runtime/auth/adapters/
  ClerkSessionTransport.ts
  ClerkTokenExchange.ts
  ClerkSessionHydrator.ts

Responsibilities:

ClerkTokenExchange:
- exchange Google ID token for backend session
- normalize Clerk responses
- normalize auth failures
- return runtime-safe session payloads

ClerkSessionHydrator:
- hydrate runtime session state from persisted metadata
- validate refresh capability
- validate expiry
- validate backend continuity

ClerkSessionTransport:
- isolated low-level Clerk communication
- NO React dependency
- NO Clerk hook dependency
- NO UI dependency

IMPORTANT:

Native auth runtime must NOT require:
- useAuth()
- useSignIn()
- useClerk()
- ClerkProvider lifecycle timing

Operationally.

---

# REMOVE REACT LIFECYCLE DEPENDENCY

Current architecture still partially depends on:

React mounting order.

This is forbidden.

Requirements:

- auth boot must complete independently
- auth restoration must complete independently
- refresh scheduling must complete independently
- lifecycle recovery must complete independently

React becomes:
a passive observer only.

---

# BOOT BARRIER HARDENING

Current RuntimeBootBarrier uses timeout continuation.

This is unsafe.

Refactor:

Boot must terminate ONLY into explicit states:

- AUTHENTICATED
- UNAUTHENTICATED
- DEGRADED
- OFFLINE_RECOVERY
- FAILED

Never:
“continue anyway after timeout.”

If timeout occurs:
- emit diagnostics
- transition FSM explicitly
- expose degraded capability state

Never leave boot ambiguous.

---

# SESSION RESTORATION VALIDATION

Current restoration appears metadata-based.

This is insufficient.

Implement:

SessionRestorationValidator.ts

Responsibilities:

- validate persisted session integrity
- validate refresh viability
- validate expiry correctness
- validate backend continuity
- validate auth confidence

IMPORTANT:

Restoration success requires:
- recoverable session continuity

NOT merely:
- persisted metadata exists

---

# SECURE STORAGE HARDENING

Current implementation uses:

@capacitor/preferences

This is insufficient for production auth guarantees.

Replace with:

- Android Keystore-backed encrypted storage
- iOS Keychain-backed encrypted storage

Create:

SecureSessionVault.ts

Requirements:

- encrypted persistence
- integrity validation
- corruption detection
- versioned session schemas
- migration-safe persistence

Never store:
- raw refresh tokens insecurely
- sensitive session material unencrypted

---

# REFRESH OWNERSHIP ISOLATION

IMPORTANT:

RuntimeCore must become:
the ONLY refresh authority.

Audit entire codebase for:

- hidden Clerk auto-refresh
- duplicate token refresh logic
- frontend session refresh ownership
- browser-cookie-based refresh semantics

Remove all duplicate refresh ownership.

Requirements:

- one refresh scheduler
- one refresh executor
- one refresh retry policy
- one refresh state authority

Refresh behavior must become:
fully deterministic.

---

# LIFECYCLE RECOVERY HARDENING

Strengthen:

AuthLifecycleCoordinator.ts

Requirements:

On:
- app resume
- process recreation
- Android activity recreation
- network reconnect
- suspend/resume cycles

The runtime must:
- reconcile auth validity
- reconcile token freshness
- prevent refresh storms
- prevent duplicate recoveries
- prevent recursive refresh loops

Implement:

LifecycleRecoveryLock.ts

Requirements:

- single recovery operation at a time
- lifecycle-safe locking
- cancellation-safe recovery
- recovery deduplication

- OFFLINE_ONLY

- INVALID

- Clerk frontend hooks are no longer operational dependencies

- process death recovery is deterministic
- auth restoration is verifiable
- lifecycle recovery is deterministic
- secure encrypted persistence exists
- auth confidence becomes explicit
- boot outcomes become deterministic
- auth logs become trace-correlated
- duplicate auth operations become impossible
- refresh storms become impossible
- offline recovery becomes deterministic
- Android lifecycle no longer breaks auth
- WebView/browser state no longer affects correctness

- a resilient mobile identity runtime
- a deterministic operational state machine

NOT:
