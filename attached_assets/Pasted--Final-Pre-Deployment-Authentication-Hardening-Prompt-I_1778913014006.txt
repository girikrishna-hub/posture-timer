# FIX DEGRADED → AVAILABLE RECOVERY GAP

IMPORTANT:

Current known issue:

- runtime can enter DEGRADED
- Clerk transport later becomes AVAILABLE
- no automatic recovery path executes

Implement recovery triggering inside:

AuthLifecycleCoordinator

Requirements:

Subscribe to capability changes:

- CLERK_RUNTIME_DELAYED → AVAILABLE
- CLERK_RUNTIME_TIMEOUT → AVAILABLE
- CLERK_RUNTIME_UNAVAILABLE → AVAILABLE

When triggered:

enqueue a guarded recovery operation ONLY if:

- confidence < VERIFIED
- no active recovery operation exists
- no active refresh chain exists
- runtime state is DEGRADED or OFFLINE_RECOVERY

Requirements:

- debounced execution
- operation-queue-controlled
- idempotent
- deterministic termination
- no recovery storms
- no duplicate refresh chains

IMPORTANT:

Transport availability does NOT imply authentication success.

Recovery must still validate:

- session continuity
- refresh viability
- token validity
- restoration integrity

before transitioning confidence/state.
