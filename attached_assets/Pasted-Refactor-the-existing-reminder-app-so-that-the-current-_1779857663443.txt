OBJECTIVE

Replace the current hardcoded sit/stand logic with a reusable engine that supports any alternating two-state protocol.

Examples the architecture must support later:
- Sit ↔ Stand
- Hold ↔ Release
- Drink Water ↔ Wait
- Bladder Empty ↔ Hydrate
- Ice On ↔ Ice Off
- Exercise ↔ Rest
- Focus ↔ Break
- Medication On ↔ Medication Off

For now, only expose:
Sit ↔ Stand

REQUIRED ARCHITECTURE

1. Define protocol model

Create a protocol definition type named TwoStateProtocol with:

- id
- name
- stateA
- stateB
- defaultDurationA
- defaultDurationB
- notificationTitleA
- notificationTitleB
- notificationBodyA
- notificationBodyB
- iconA
- iconB

2. Define runtime state model

Create ProtocolSession with:

- active protocol
- current state
- current phase start time
- phase duration
- paused status
- remaining time
- session status

3. Create protocol engine

Extract timer execution logic into ProtocolEngine

Responsibilities:
- start()
- pause()
- resume()
- stop()
- transition()
- computeRemaining()
- persist()
- restore()
- emit notifications
- emit state transitions

The engine must be protocol-agnostic.

It must not contain sit/stand-specific logic.

4. Adapter layer for backward compatibility

Create SitStandProtocolAdapter

This adapter maps current sit-stand UI interactions into the generic engine.

This ensures all current screens continue to function unchanged.

5. Preserve persistence compatibility

Existing saved sit/stand session data must continue to restore correctly.

If needed:
- implement migration logic
- detect old persisted schema
- transform to new protocol schema

No user data loss.

6. Notification abstraction

Generalize notifications so they are generated from protocol state definitions.

Current sit/stand notifications must render exactly as before.

7. Future extensibility hooks (internal only)

Add infrastructure for:

- registerProtocol()
- getProtocolById()
- listProtocols()

Do NOT expose this in UI yet.
